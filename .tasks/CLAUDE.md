## Рабочий процесс

### После каждого изменения кода — запускать сборку:
```bash
npm run build
```
Делать это самостоятельно, без просьбы. Если сборка упала — разобраться и починить.

### Коммиты:
- Могу коммитить самостоятельно, **не пушу без подтверждения**
- Сообщение коммита на русском, кратко

### Задачи (все папки внутри `.tasks/board/`):
- Новые идеи → `board/backlog/`
- Взятые в работу → `board/in-progress/`
- Готово → `board/done/`
- Баги и исправления → `board/fix/`
- Закрытые/отменённые → `board/closed/`
- Файл `.tasks/IMPLEMENTED.md` — обновлять после реализации фичи
- **Никогда не переиспользовать ID задач** — всегда брать следующий номер выше максимального существующего (проверяй все папки board/)

## Правила написания кода

1. **Перед изменением — читать файл**, не угадывать структуру
2. **Не рефакторить то, что не просили** — только нужное
3. **Не добавлять комментарии** к коду который не менял
4. **Негативные эффекты в чате** — всегда красный цвет `#e74c3c`
5. **Бонусы в чате** — голубой `#aed6f1` (STR), фиолетовый `#a29bfe` (стелс), зелёный `#69f0ae` (прицел)
6. **GM-модель** — логика изменения состояния только на клиенте ГМа через socketlib

## После реализации — инструкция по тестированию

После каждой реализованной механики писать краткую инструкцию как проверить что она работает:
- Что подготовить (статусы, предметы, акторы)
- Что сделать (конкретные шаги)
- Что ожидать увидеть (в чате, на листе, в консоли)
- Граничные случаи — что НЕ должно срабатывать

---

## Что проверять при реализации механик

Перед реализацией — свериться с `GAME_RULES.md`:
- Порядок применения множителей урона
- Какие штрафы стакаются, какие перемножаются
- Что игнорирует DR, а что нет

## Архитектурные особенности системы

- **FoundryVTT v13+** — использовать актуальное API (не устаревшее)
- Зависимость: **socketlib** (обязательный модуль)
- Сборщик: **rollup** → выход `dist/zsystem.js`
- Локализация: русский (`lang/ru.json`) — основной язык
